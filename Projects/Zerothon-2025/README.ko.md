## 🎯 프로젝트 개요

**"우리팀 뭐함?"** - 이 질문에서 시작된 프로젝트입니다.

Blitz-Q는 팀원들의 일일 보고서를 AI가 자동으로 분석하여 팀 상태를 한눈에 파악할 수 있게 해주는 협업 도구입니다. 리그 오브 레전드의 블리츠크랭크 Q스킬처럼, 모든 팀원의 진행상황을 한 번에 잡아냅니다.

**대회 정보**:
- **규모**: 전국 단위 해커톤 (단국대학교 제로톤 2025)
- **참가팀**: 총 80명, 19개 팀
- **수상**: 최우수상 (2등)
- **개발 기간**: 2025.03.26 ~ 2025.03.29 (100시간)
- **심사위원**: Toss, Naver, 금융공기업 현직자

## 🛠 기술 스택

### Frontend (담당 영역)
- **Vanilla JavaScript** - 프레임워크 없이 순수 JS 구현
- **HTML5 + CSS3** - 반응형 디자인

### Backend (팀원 담당)
- **Node.js + Express.js**
- **SQLite3**
- **JWT 인증**

### AI & APIs
- **Google Gemini API** - AI 보고서 분석
- **Google OAuth 2.0** - 소셜 로그인

## 💻 내 역할

### 1️⃣ 프론트엔드 개발

**메인 대시보드 (`index.html`)**:
- 팀원별 진행률, To-Do 리스트, AI 분석 결과를 한 화면에 배치
- 직관적인 UI/UX 설계 (진행률 프로그레스 바, 색상 기반 상태 구분)

**보고서 작성 페이지 (`report.html`)**:
- "어제 한 일", "오늘 할 일", "이슈사항" 입력 폼 구조화
- 동적 할일 체크리스트 (추가/삭제 가능)

**전체 프론트엔드 디자인**:
- 다른 팀원과 협업하여 전체 UI/UX 담당
- 핵심 철학: "사용자 친화적인 UI/UX" (능력 과시 X)
- Google 로그인 버튼처럼 익숙한 패턴 적극 활용

**기술적 제약과 학습**:
- 첫 실전 프로젝트 - React, Vue 등 프레임워크 경험 없음
- 순수 JS/HTML/CSS만으로 동적 UI 구현
- 프레임워크 없이도 가능하지만, 효율성 차이를 체감 → 이후 React 학습 시작

---

### 2️⃣ 팀장 역할

**팀 구성**: 프론트엔드 3명 + 백엔드 2명 (실력 차이 큼)

**역할 맞춤형 분배**:
- 실력 있는 팀원: 핵심 기능 개발 집중
- 초보 팀원: 자료조사, 디자인 초안 등 성장 가능한 역할
- 각자의 강점을 살리면서 모두가 기여할 수 있게 조율

**갈등 조율**:
- 상황: 제로톤 전날 회의 시간 조율 과정에서 감정 마찰 발생
- 해결: 개별 의견 청취 → 객관적 기준 재조정 → 팀 목표 상기

**리더십 철학**: "앞에서 당기는 것이 아니라 뒤에서 미는 것"

---

### 3️⃣ 발표자 역할

**발표 순서**: 19팀 중 18번째 (매우 불리한 순서)

**발표 전략**:
1. 오프닝: "우리팀 뭐함?" - 세 글자로 공감대 형성
2. 핵심 메시지: 문제 공감 → 솔루션 제시 → 실제 구동 시연
3. 발표 철학: "최대한 많은 청중이 최대한 많은 시간동안 내 이야기를 듣게 하는 것"

**결과**: 발표 점수 1등 획득

## 🤝 팀 협업과 Git

**현재 커밋 히스토리 상황**:
- 한 팀원이 `final commit`으로 모든 코드 일괄 업로드
- 개별 커밋 기록 거의 없음 (나는 README 작성 정도만 기록)

**실제 개발 과정**:
- 로컬 개발 → 오프라인 통합 → 최종 일괄 커밋
- 당시 Git 브랜치 전략 미숙

**실제 기여 내용**:
- 메인 대시보드 및 보고서 작성 폼 UI/UX 설계/구현
- 전체 프론트엔드 디자인 방향 설정
- 팀 일정 조율 및 역할 분배
- 최종 발표 및 PPT 제작

**학습 포인트**: 팀 협업 시 Git 브랜치 전략 필수 → 이후 프로젝트에서 PR 기반 협업 적용

## 💡 배운 점

### 기술적 학습

**프레임워크의 필요성**:
- 순수 JS로 동적 UI 구현 경험
- DOM 조작/이벤트 핸들링을 일일이 작성하며 비효율성 체감
- 이후 React 학습 시작

**Git 협업의 중요성**:
- 브랜치 전략, Pull Request, 커밋 메시지 컨벤션 학습

**디자인의 중요성**:
- 아무리 좋은 기능도 UI/UX가 복잡하면 외면받음
- 익숙한 패턴 활용의 중요성

---

### 리더십과 협업

**뒤에서 미는 리더**:
- 앞에서 끌면 팀원이 수동적, 뒤에서 밀면 주도적
- 능력 부족한 팀원에게도 의미 있는 역할 부여

**미움 받을 용기**:
- 팀 방향이 잘못되면 리더가 나서야 함
- 모두를 만족시킬 수는 없지만 팀 목표는 지켜야 함

---

### 발표 스킬

**청중 집중 유지**:
- 오프닝 임팩트: 3글자 공감대 형성
- 스토리텔링: 문제 → 솔루션 → 시연
- 완성된 제품 직접 시연

---

### 학습 방식의 전환

**Bottom-up → Top-down**:
- 이전: 이론부터 차근차근 (느리고 지루함)
- 이후: 실제 제품을 만들면서 부족한 부분 찾기 (빠르고 즐거움)

**완성의 중요성**:
- 80% 아이디어보다 100% 완성된 제품
- 실제 사용 가능한 수준까지 구현해야 인정받음

**우리 팀이 상을 받은 이유**:
1. 완성도: 프론트-백엔드 완전 연동
2. 발표력: 청중을 사로잡는 스토리텔링
3. 실용성: 실제로 쓸 수 있는 서비스

## ❓ 면접 예상 질문

### Q1: Git 커밋 기록에 본인 기여가 거의 없는데, 실제로 개발한 건가요?

**A**: 네, 맞습니다. 당시 오프라인 협업 중심으로 개발했고 Git 브랜치 전략을 몰라서 한 팀원이 최종 코드를 일괄 커밋했습니다. 실제로는 메인 대시보드와 보고서 작성 폼의 UI/UX를 설계/구현했고, 팀장으로 일정 조율 및 최종 발표를 담당했습니다. 이 경험으로 Git 협업의 중요성을 깨닫고 이후 프로젝트에서는 브랜치 전략과 PR 기반 협업을 적용했습니다.

---

### Q2: 프레임워크 없이 순수 JS로만 개발한 이유는?

**A**: 첫 실전 프로젝트라 프레임워크를 몰랐기 때문입니다. DOM 조작과 이벤트 핸들링을 직접 구현하면서 JavaScript 동작 원리를 깊이 이해했지만 코드 재사용성과 유지보수 한계를 느꼈습니다. 프레임워크가 왜 필요한지 체감한 후 바로 React 학습을 시작했고, 현재는 효율적으로 개발하면서도 순수 JS 경험 덕분에 내부 원리를 이해하고 있습니다.

---

### Q3: 팀장으로서 가장 어려웠던 점과 해결 방법은?

**A**: 팀원 간 실력 차이가 매우 컸던 것이 가장 어려웠습니다. 실력 있는 팀원에게는 핵심 기능 개발을, 초보 팀원에게는 자료조사나 디자인 초안 같은 의미 있는 역할을 부여했습니다. 리더는 "앞에서 당기는 것이 아니라 뒤에서 미는 것"이라는 것을 배웠고, 각자의 강점을 살리고 약점은 팀으로 보완하는 것이 핵심임을 깨달았습니다.

---

### Q4: 발표에서 "우리팀 뭐함?"이라는 오프닝을 선택한 이유는?

**A**: 19팀 중 18번째 발표라 청중의 집중력이 최저점일 것이라 예상했습니다. 모든 팀 프로젝트에서 겪는 문제를 3글자로 요약해 공감대를 형성하고, 짧고 강렬한 질문으로 주의를 집중시켰습니다. 결과적으로 발표 점수 1등을 받았고, 심사위원들이 "기억에 남는 발표"라고 평가해주셨습니다.

---

### Q5: 이 프로젝트를 통해 가장 크게 성장한 부분은?

**A**: 세 가지 영역에서 성장했습니다. 첫째, 기술적으로는 순수 JS로 프론트엔드를 구현하며 프레임워크의 필요성을 깨달아 이후 React, Django로 기술 스택을 확장했습니다. 둘째, 협업 능력으로는 Git 전략의 중요성과 능력 차이가 있는 팀을 이끄는 법을 배웠습니다. 셋째, 학습 방식이 Bottom-up(이론 먼저)에서 Top-down(프로젝트 먼저)으로 전환되어 훨씬 효율적이고 즐겁게 배우게 되었습니다.
