## 📌 핵심 태도

### 1. 복잡한 문제는 쪼개서 접근하라

**태도**: "한 번에 다 하려고 하지 말자"

#### 나쁜 습관
```
"크롤링도 하고 DB도 저장하고 API도 만들고 프론트도 다 한 번에!"
→ 어디서 막혔는지 모름
→ 디버깅 지옥
→ 성취감 없음
```

#### 좋은 습관
```
1. 크롤링만 먼저
   → TXT 파일로 저장해서 데이터 확인
   → "이 단계는 됐다!" ✅

2. DB 저장
   → 쿼리로 확인
   → "DB 저장도 됐다!" ✅

3. API 구축
   → Postman으로 테스트
   → "API도 됐다!" ✅

4. 프론트 연동
   → 최종 완성! ✅
```

**Dipping 프로젝트에서 배운 것**:
- jQuery UI Datepicker가 안 될 때도 쪼갰음
- "Datepicker 열기 → 날짜 클릭 → Apply 버튼" 하나씩 테스트
- 각 단계가 성공해야 다음으로

**앞으로 개발할 때**:
- 막막한 기능 만나면 → 종이에 단계 적기
- 한 단계씩 완성하고 확인
- 성취감 쌓으면서 진행

### 2. AI는 도구다, 내가 운전자다

**태도**: "AI한테 의존하지 말고, AI를 활용하자"

#### 나쁜 습관
```
나: "크롤러 만들어줘"
AI: (코드 300줄)
나: (복붙) → 실행 → 에러 → "왜 안 돼?" → 다시 AI한테
```

#### 좋은 습관
```
나: "Selenium으로 jQuery UI Datepicker 조작이 안 돼.
     value를 설정해도 내부 상태가 업데이트 안 됨.
     시도한 방법: 1) JS로 value 설정 2) send_keys
     둘 다 실패. 단계별 방법 알려줘."

AI: (구체적인 답변)

나: (코드 읽고 이해) → (테스트) → (안 되면 다시 질문)
```

**Dipping 프로젝트에서 배운 것**:
- AI 코드 무작정 복붙 → 산으로 감
- AI가 준 코드 이해하고 테스트 → 성공
- "왜 이렇게 했을까?" 생각하는 습관

**앞으로 개발할 때**:
- AI한테 질문할 땐 구체적으로 (문제, 시도, 원하는 결과)
- AI 코드는 반드시 읽고 이해하고 테스트
- "이해 못 하면 복붙 금지"

### 3. 에러는 적이 아니라 선생님이다

**태도**: "에러 나도 당황하지 말자"

#### 나쁜 습관
```
[에러 발생]
→ "아 짜증나" → 포기
→ "AI한테 물어볼까" (바로)
```

#### 좋은 습관
```
[에러 발생]
→ "오케이, 뭐가 문제지?" (냉정)
→ 에러 메시지 정확히 읽기
→ 구글링 (에러 메시지 복사)
→ 안 되면 문제 쪼개기
→ 그래도 안 되면 AI/팀원한테 구체적으로 질문
```

**Dipping 프로젝트에서 배운 것**:
- jQuery UI 실패 → "왜 안 될까?" → 내부 동작 이해
- "값 설정 ≠ 사용자 액션" 깨달음
- 에러가 더 깊은 이해로 이어짐

**앞으로 개발할 때**:
- 에러 메시지부터 정확히 읽기
- 구글링으로 비슷한 케이스 찾기
- 에러 날 때마다 배운다는 마음

### 4. 완벽보다 완성을 먼저

**태도**: "일단 동작하게 만들고 개선하자"

#### 나쁜 습관
```
"완벽한 설계를 해야 해..."
→ 1주일 고민
→ 시작도 못 함
→ 데드라인 다가옴
```

#### 좋은 습관
```
1일차: 일단 동작하는 코드 작성
2일차: 리팩토링 (함수 분리, 이름 정리)
3일차: 최적화 (성능 개선)
→ 완성!
```

**Dipping 프로젝트에서 배운 것**:
- 처음엔 print()로 디버깅 → 나중에 logging으로
- 처음엔 하드코딩 → 나중에 설정 파일로
- 일단 완성하니 개선 포인트가 보임

**앞으로 개발할 때**:
- 설계보다 실행 먼저
- "일단 돌아가게" → "이쁘게" → "빠르게"
- 완벽 추구로 시작 못 하는 것보다, 완성하고 개선하는 게 낫다

### 5. 전체 그림을 이해하라

**태도**: "숲을 보고 나무를 보자"

#### 나쁜 습관
```
"내가 짠 코드만 알면 돼"
→ 문제 생기면 어디서 막혔는지 모름
→ 디버깅 오래 걸림
```

#### 좋은 습관
```
[크롤링] → [DB] → [API] → [프론트] → [사용자]

문제: "프론트에 데이터 안 나옴"

→ 크롤링: TXT 확인 → ✅ 데이터 있음
→ DB: 쿼리 → ✅ 저장됨
→ API: Postman → ❌ 500 에러!
→ 발견: serializer 설정 오류

→ 빠른 해결!
```

**Dipping 프로젝트에서 배운 것**:
- 데이터 흐름 이해 → 문제 빠르게 찾음
- 각 단계 입출력 알기
- 최적화 포인트 발견

**앞으로 개발할 때**:
- 새 프로젝트 시작하면 전체 흐름 먼저 그리기
- 각 단계에서 "입력 뭐고, 출력 뭐지?" 파악
- 문제 생기면 흐름 따라 하나씩 체크

### 6. 막히면 구글링 → 쪼개기 → 질문

**태도**: "혼자 고민만 하지 말고, 단계적으로 도움 받자"

#### 해결 순서
```
1. 에러 메시지 정확히 읽기
   → 어떤 에러인지 파악

2. 구글링 (에러 메시지 복사)
   → 비슷한 케이스 찾기
   → StackOverflow, 공식 문서

3. 문제 쪼개기
   → 최소 재현 코드 만들기
   → "어느 부분에서 막혔나?"

4. AI/팀원한테 질문
   → 문제 상황 + 시도한 방법 + 원하는 결과
   → 구체적으로!
```

**Dipping 프로젝트에서 배운 것**:
- jQuery UI 막힘 → 구글링 → 공식 문서 → AI 질문
- "막혔다" → 바로 AI (X)
- "이해하고 넘어가자" (O)

**앞으로 개발할 때**:
- 30분 고민해도 안 되면 검색
- 1시간 검색해도 안 되면 질문
- 혼자 끙끙대는 시간 줄이기

### 8. if/else vs try/except 구분하라

**태도**: "예상된 상황과 예상 못한 에러를 구분하자"

#### if/else (예상된 조건)
```python
# 논리적 갈림길
if country == "US":
    filter_us_events()
else:
    filter_all_events()
```

#### try/except (예상 못한 에러)
```python
# 실패할 수 있는 작업
try:
    data = crawl_website()
except NetworkError:
    logger.error("네트워크 오류")
    retry()
```

**Dipping 프로젝트에서 배운 것**:
- 크롤링은 실패 가능성 높음
- try/except 없으면 한 번 에러로 전체 중단
- 안정성 확보

**앞으로 개발할 때**:
- 네트워크, 파일 IO, 외부 API → try/except
- 조건 분기 → if/else
- "이거 실패할 수 있나?" 생각하는 습관

## ❓ 면접 예상 질문

**Q1: 복잡한 문제를 만났을 때 어떻게 접근하나요?**
> A: 큰 문제를 작은 단계로 쪼갭니다. Dipping 프로젝트에서 "크롤링해서 캘린더에 띄우기"를 ①크롤링 ②DB 저장 ③API ④프론트로 나눴습니다. 각 단계마다 검증하여 문제 범위를 좁히고, 성취감도 느낄 수 있었습니다.

**Q2: AI를 개발에 어떻게 활용하나요?**
> A: AI는 도구이고 제가 운전자입니다. 문제 상황과 시도한 방법을 구체적으로 질문하고, AI가 제안한 코드는 반드시 이해하고 테스트합니다. 코드 이해 없이 복붙하면 나중에 유지보수가 어렵기 때문입니다.

**Q3: 막혔을 때 어떻게 하나요?**
> A: 먼저 에러 메시지를 정확히 읽고 구글링합니다. 30분 안에 해결 안 되면 문제를 더 작게 쪼개서 최소 재현 코드를 만듭니다. 그래도 안 되면 구체적으로 질문합니다. 혼자 끙끙대는 시간을 줄이려 노력합니다.

**Q4: 에러를 어떻게 받아들이나요?**
> A: 에러는 배움의 기회입니다. jQuery UI Datepicker 조작이 계속 실패했지만, 원인을 파고들며 "값 설정 ≠ 사용자 액션"이라는 핵심을 배웠습니다. 에러를 두려워하기보다 "왜 이렇게 됐지?"라고 접근합니다.

**Q5: 완벽한 코드를 추구하나요?**
> A: 완벽보다 완성을 먼저 합니다. Dipping 프로젝트에서 처음엔 print()로 디버깅했지만, 일단 동작하게 만든 후 logging으로 개선했습니다. 완벽 추구로 시작 못 하는 것보다, 완성하고 개선하는 게 낫다고 배웠습니다.
